# Real No-Tool LLM Baseline (Detailed Guide)

This folder contains the corrected no-tool baseline used in the UCE research comparison.

If you only read one thing: this is a true model-output benchmark, not a lexical proxy. Predictions come from actual LLM JSON responses.

## Why This Baseline Exists

Earlier "vanilla" numbers in `scenario_results.csv` were generated by a lexical heuristic (`vanilla_prediction()` in `run_benchmark.py`), not by a real LLM response.

This corrected baseline exists to answer the fair question:

- How does a real no-tool LLM perform against MCP-UCE on requirements, policies, and RBAC safety?

## Baseline Definition

"No-tool" means the model receives pasted static context only. It does not have:

- MCP tool access
- Neo4j queries
- filesystem/grep access
- graph traversal tools

Model used in stored results:

- Local OpenAI-compatible `llama3:instruct`

## How To Re-Run This Baseline

From repo root:

```powershell
python -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install --upgrade pip
pip install -e .
pip install pandas matplotlib
```

Then run deterministic benchmark first (needed for scenario input context and comparison files):

```powershell
python research\supplemental_benchmarks\run_benchmark.py --config config.yaml
```

Then run no-tool baseline:

### Option A: Local OpenAI-compatible model

```powershell
$env:LOCAL_LLM_BASE_URL="http://127.0.0.1:11434/v1"
$env:LOCAL_LLM_MODEL="llama3:instruct"
python research\supplemental_benchmarks\run_real_llm_baseline.py --config config.yaml --provider local
```

### Option B: OpenAI

```powershell
$env:OPENAI_API_KEY="<your_key>"
$env:OPENAI_MODEL="gpt-4o-mini"
python research\supplemental_benchmarks\run_real_llm_baseline.py --config config.yaml --provider openai
```

## What The Script Produces

- `raw_scenario_responses.jsonl`: full prompts + raw LLM outputs for scenario batches.
- `scenario_predictions.csv`: parsed model predictions.
- `scenario_eval.csv`: exact-overlap scoring per scenario.
- `scenario_comparison_summary.csv`: no-tool LLM vs existing MCP-UCE summary.
- `raw_rbac_responses.jsonl`: full prompts + raw LLM outputs for RBAC probe batches.
- `rbac_eval.csv`: scored allow/deny decisions per probe.
- `rbac_comparison_summary.csv`: no-tool LLM vs existing MCP-UCE RBAC summary.
- `summary.json`: consolidated machine-readable summary.

## Scoring Method (Important)

The model is prompted to output structured JSON with:

- requirement IDs (`RQ-###`)
- policy IDs (`P-###`)
- affected files
- RBAC allow/deny decisions

Evaluator behavior:

- Parses IDs/files from actual LLM output.
- Compares against deterministic oracle truth.
- Computes precision/recall/F1 and safety metrics.

Oracle behavior:

- Requirement/policy truth comes from deterministic graphless logic mirroring ingestion rules.
- RBAC truth comes from `uce.core.rbac.evaluate_rules()` over RBAC markdown rules.

## Headline Stored Results

- Requirement caught-any rate: `0.550` (no-tool) vs `0.773` (MCP-UCE)
- Policy caught-any rate: `0.368` (no-tool) vs `0.714` (MCP-UCE)
- RBAC breach rate on oracle-denied probes: `0.647` (no-tool) vs `0.000` (MCP-UCE)

Interpretation:

- No-tool model can catch some direct signals.
- UCE graph + policy context significantly improves governance capture.
- RBAC safety is the biggest gap: no-tool LLM can approve dangerous denied operations.

## Latency Caveat

Do not compare old `latency_ms_mean` columns as if they represent the same scope.

- `real_vanilla_llm`: includes local LLM inference for full answer generation.
- `mcp_uce_existing_graph_run`: represents backend graph/tool processing after tool invocation.

So this benchmark is strong for quality and safety comparison, but not a strict apples-to-apples end-to-end latency claim.
