ID: RQ-007
Title: Account OAuth credentials must be stored securely and scoped
Priority: Critical
Description: The `accounts` table MUST store `access_token` and `refresh_token` for OAuth 2.0 provider flows. Both tokens are highly sensitive credentials; their storage, rotation, and deletion must follow least-privilege and secure-storage principles.
Acceptance Criteria:
  - Tokens are stored encrypted at rest (AES-256 or provider-native vault; NOT as plaintext columns).
  - `access_token` has an expiry column (`access_token_expires_at`) and is never used past its expiry.
  - `refresh_token` rotation: after each use the old token is invalidated and the new one stored atomically.
  - Token scopes are stored alongside the tokens (`scope` column); code MUST NOT request broader scopes than declared.
  - Tokens are revoked at the provider on account deletion; a best-effort revocation call is made before the DB row is deleted.
  - No token value is ever written to application logs or error reports.
Tags: oauth, credentials, security, encryption
GDPR Note: OAuth tokens are personal data (Art. 4(1)) because they grant access to the user's account at the provider. They must be included in Art. 17 erasure requests (revoke and delete). Art. 32 requires adequate technical measures (encryption at rest, access controls).
