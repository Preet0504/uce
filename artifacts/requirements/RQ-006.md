ID: RQ-006
Title: Sessions require tokens and expiry
Priority: High
Description: Every record in the `sessions` table MUST include `token` (or equivalent session identifier), `expires_at`, and a reference to the owning user (`user_id`). Sessions are short-lived credentials; their lifecycle must be tightly controlled to limit the blast radius of token theft.
Acceptance Criteria:
  - `token` is a cryptographically random value (≥128 bits of entropy); it MUST NOT be a sequential integer or derived from predictable data.
  - `expires_at` is set on creation and enforced on every authenticated request; expired sessions are rejected with HTTP 401.
  - A background job purges expired session rows within 1 hour of expiry.
  - Session invalidation on logout MUST delete the row (not merely set a `revoked` flag) to prevent database enumeration.
  - Sessions MUST be bound to the originating IP or device fingerprint when the product risk model requires it.
Tags: authentication, sessions, security
GDPR Note: Session tokens are pseudonymous personal data (Art. 4(1) / Recital 26) because they can be linked back to a specific user. They must be included in any Art. 17 erasure response and must not be retained after the user's account is deleted.
