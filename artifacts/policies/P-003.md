ID: P-003
Title: User Identity and Session Security Policy
Type: GDPR Art. 32 / Internal Security
Description: User identity records and session tokens must remain verifiable, unpredictable, and short-lived. Any code change that modifies how email addresses, session tokens, or password hashes are stored, compared, or transmitted must pass a security review before merge. This policy exists to prevent credential theft, session fixation, and data exposure to unauthorised parties.
Controls:
  - C1: `email` column has a UNIQUE constraint; duplicate registration attempts are rejected with a non-enumerable error message (to prevent user enumeration).
  - C2: Session `token` values are generated by a CSPRNG; minimum 128 bits entropy; stored as hashed (bcrypt/argon2) or as an opaque reference to an encrypted store.
  - C3: `expires_at` is enforced at the middleware layer before any authenticated route handler executes.
  - C4: Sessions are invalidated (row deleted) immediately on explicit logout; a user can invalidate all sessions via "logout all devices".
  - C5: Password hashes are NEVER returned in API responses; SQL queries that JOIN `users` must explicitly exclude the hash column.
Enforces: RQ-005, RQ-006
Enforcement Mode: block
GDPR Relevance: Email (Art. 4(1) direct identifier, high sensitivity) and session tokens (pseudonymous personal data) are governed by this policy. Unauthorised access to these fields constitutes a personal data breach under Art. 4(12) with a potential 72-hour notification obligation under Art. 33.
Review Date: 2026-12-01
Approver: Security Lead, Data Protection Officer
